# 슬랙봇 VM 보안 강화 방안

이 문서는 GCP에 배포된 슬랙봇 VM의 보안을 강화하기 위한 방법을 설명합니다.

## 현재 아키텍처의 보안적 이점 (Socket Mode)

현재 우리 봇은 **Socket Mode** 방식을 사용하고 있으며, 이는 일반적인 웹훅(Webhook) 방식보다 뛰어난 보안적 이점을 가집니다.

- **웹훅(Webhook) 방식:** 봇이 외부에서 접근 가능한 공개 주소(URL)를 가지고, 슬랙 서버가 그 주소로 HTTP 요청을 보내는 방식입니다. 이 경우, 반드시 특정 포트(예: 443)를 외부에 열어두어야 합니다.
- **소켓 모드(Socket Mode) 방식 (현재 방식):** 봇이 먼저 슬랙 서버로 **나가는(outbound) 연결**을 시작하여 웹소켓(WebSocket) 터널을 만듭니다. 슬랙은 이 이미 연결된 터널을 통해 이벤트를 보내줍니다.

결론적으로, 현재 우리 VM은 슬랙으로부터 **들어오는(inbound) 연결을 받을 필요가 전혀 없습니다.** 봇이 작동하기 위해 외부에 포트를 열어둘 필요가 없다는 의미이며, 이는 매우 안전한 구조입니다.

## 핵심 보안 강화 지점: SSH 접근 통제

진짜 보안 강화가 필요한 지점은 개발 및 유지보수를 위한 **SSH 접근**입니다. 현재 VM은 `gcloud compute ssh` 명령어로 접속이 가능하도록 SSH 포트(22)가 외부에 열려있을 수 있습니다. 이 접근을 아무나 할 수 없도록 통제하는 것이 VM 보안의 핵심입니다.

가장 안전하고 현대적인 방법은 **IAP(Identity-Aware Proxy)**를 사용하는 것입니다.

### IAP (Identity-Aware Proxy)란?

IP 주소 기반이 아닌, **Google 계정의 ID를 기반으로** VM 접근 권한을 제어하는 GCP의 제로 트러스트(Zero-Trust) 보안 서비스입니다. IAP를 사용하면 VM의 SSH 포트를 인터넷에 직접 노출하지 않고도, 허가된 사용자만 Google의 프록시를 통해 안전하게 접속할 수 있습니다.

## 보안 강화 실행 계획

1.  **기존 방화벽 규칙 삭제:** 현재 SSH 접속(`tcp:22`)을 허용하는, 외부에 열린 방화벽 규칙을 찾아 삭제합니다.
2.  **IAP용 방화벽 규칙 생성:** Google의 IAP 프록시 서비스로부터의 SSH 연결만 허용하는 새로운 방화벽 규칙을 만듭니다.
3.  **사용자에게 IAP 터널링 권한 부여:** 특정 Google 계정(예: 개발자 본인)이 IAP를 통해 VM에 접속할 수 있도록 IAM에서 `IAP-secured Tunnel User` 역할을 부여합니다.
